第四个系统对象
副标题： agent 是在电脑上做事；Neural Computer 是让电脑本身学会做事，并把能力变成运行时对象。
引言
过去几十年，人类通过计算机做事；过去几年，AI 也开始通过计算机做事。下一个问题是：AI 能不能不只是使用计算机，而是自己成为一种可靠的通用计算机？
神经计算机（Neural Computer，NC）讨论的，正是这一点：模型不再只是生成答案、使用工具或推演世界，而开始承担一台计算机最核心的运行时职责。
今天，agent 已经进入了一部分生产流程，也已经能调工具、跨界面、串任务；但越往深处推，同一个问题就会反复浮出来：能力装不住、运行会漂、更新不可追责。
NC 要解决的，不是让 agent 更会做事，而是第一次把能力安装、行为一致性和显式更新写进运行时契约。 它不是更强的 agent，也不是计算机环境里的 world model，更不是传统计算机外挂了一个智能层。它争夺的，是“运行中的机器”这一层。

1）为什么是现在：第四个系统对象开始有了轮廓
过去几年，三件事同时发生。
第一，agent 变得越来越会做事。
 系统已经不再只是回答问题，而是开始调用工具、操作界面、串联流程，逐步进入真实工作流。很多团队面对的，已经不是“agent 能不能 demo”，而是“agent 能不能稳定接入生产”。
第二，world model 变得越来越会推演。
 模型对动态过程的建模能力持续增强，不再只是“看到当前”，而开始在内部携带一个关于“接下来会发生什么”的动态模型。
第三，传统计算机在 AI 时代的结构性摩擦越来越明显。
 越来越多任务不是固定规则，而是开放式要求；不是短流程，而是长时程协作；不是一次性输入输出，而是持续交互。显式程序和分层软件栈仍然强大，但在很多以自然语言、示范、界面操作和弱约束为主的任务里，它们开始显得笨重：每一步都还能做，但要先把连续的做事方式翻译成离散接口、脚本和流程。
这三件事最后指向同一个问题：能否出现一种新的运行时，把执行、推演与能力沉淀统一进同一个学习系统？
沿着现有两条线继续往前，方向其实已经很清楚：agent 这条线走向递归自我改进，world model 这条线走向更深的推演—行动闭环。但它们强化的，终究还是既有对象上的系统能力：前者增强智能体，后者深化环境建模。与此同时，传统计算机本身也在为 AI 快速重设计——从芯片、编译器到系统软件，都在变得更适合模型运行；但这仍然是在既有计算范式内做底座适配，而不是产生一个新的系统对象。
我更想讨论的是另一条路：不是继续把智能体做强，也不是只让既有计算底座更适合 AI，而是把运行时本身改写成学习系统——让“电脑本身”成为可以学习的机器。
NC 真正新颖的地方在于：持久状态、状态推进、能力安装与输入输出协调，开始向同一个学习型运行时收拢。模型不再只是生成答案或调用工具，而开始承担“运行中的机器”这一角色。
由此带来一个此前不够显眼、却极关键的变化：交互本身，正在接近能力安装。自然语言指令、键鼠轨迹、屏幕状态变化与任务反馈，过去更多只是日志；在 NC 的语境里，它们开始有机会成为塑造系统行为的材料。
今天我们主要用代码安装能力；在 NC 的语境里，示范、交互轨迹与约束也可能成为安装介质。被安装和治理的对象，也会从应用和脚本，转向能力与工作方式本身。
但这条路真正的难点，从来不是 demo 够不够惊艳，而是进入长时程、多轮交互和真实部署之后，系统会不会漂。三年前问这个问题还太早；今天，第一次有了原型证据。问题不再是它像不像一个新对象，而是它离一台真正可用的机器还差什么。

2）什么是 NC，什么又算 CNC？
先把四个对象排开。
传统计算机： 对象是代码，程序栈负责执行。
 agent： 对象是任务，在现有计算机上调用工具完成任务。
 world model： 对象是环境动态，学习状态如何随时间和行动演化。
 NC： 对象是运行时，能力被安装进学习型运行时。
区别不在智能高低，而在能力到底栖居在哪里。
这里所谓“系统对象”，不是面向对象里的 object，而是系统主要围绕什么来组织能力、推进状态并承担职责。“第四个”也不是教科书目录，而是沿着这条切分轴给出的工作分类。
这四个对象的差别，不只是“谁更聪明”，而是“谁在承担系统的哪一层职责”。传统计算机负责按规则执行；world model 负责学习环境如何演化；agent 负责在现有计算机上把任务做出来；NC 则想把持久状态、能力安装、状态推进和输入输出协调一起收进学习型运行时。它不是前三者的简单叠加，而是在争夺“运行中的机器”这一层。
NC 是这个新类别的名字，CNC 是它的完成形态。
 NC 指的是这样一种系统：真正决定系统如何运行的那一层——持久状态、状态推进、能力安装与运行协调——开始主要由学习型运行时承担，而不是由外部程序栈编排。
 CNC 则是 NC 的成熟形态：它不只是能演示几个原型，而是能够长期运行、稳定复用、显式更新、可追踪治理。
一句话判断：能力装得进去、行为稳得住、更新追得到。
先看一个最具体的例子。
假设我们给系统安装一种“自动科研”能力：它能持续跟进一个课题，筛论文、记笔记、比较方法，并在你写综述时重新组织材料。第一次安装时，用户不是写一套完整流程，而是通过几次共同工作，把证据标准、取舍习惯和笔记分类教给系统；之后它应稳定沿用这套做法，除非你显式更新标准。出了问题，你还能回放它为何保留某篇论文、为何删去某条结论，并回到上一个稳定版本。
这个例子里，agent 更像一个会帮你搜论文、开网页、改文档的助手；NC 更像研究工作台本身，能力被安装在一个持续运行的环境里。
这个例子也说明：任务描述、示范、交互轨迹和约束，都可能成为能力安装的介质——但这并不意味着它们天然就等于程序。关键在于：只有当它们进入运行时，成为可复用、可追踪、可回退的对象时，才真正接近“编程”。
最容易混淆的是三件事：Prompt、fine-tuning 和长期记忆。它们都能参与能力形成，但都不自动等于安装能力。判别标准只有一个：这套做法是否以可命名、可复用、可回退的对象形式进入运行时，并能脱离特定会话被稳定调用。能，才叫安装；不能，就还是临时塑形、参数更新或历史召回。
长期在线、带记忆的 agent，已经很接近 NC 的外观：它能接任务、跨会话延续、替你操作电脑。但两者仍不是同一件事。agent 的持久状态服务于任务完成；NC 的持久状态服务于运行时本身。只要系统真正的依据仍主要在外部文件、工作流或程序栈里，模型负责调用而不负责运行时本体，它就更接近长期 agent，而不是 NC。
如果把边界再压缩一点，NC 和另外三个对象的区别，可以用三句话说清。
和传统计算机相比，NC 改写的是运行时语义：代码不再是唯一的能力入口，能力本身开始成为可安装、可复用、可版本化的对象。
 和 world model 相比，NC 改写的是系统职责：前者学习“世界如何演化”，后者要负责“机器如何运行”，重点不在预测，而在执行、编程与治理。
 和 agent 相比，NC 改写的是能力的栖居位置：agent 在现有计算机上调用能力，NC 则试图把能力安装到运行时本身。
更深一层地看，NC 与传统计算机的差别至少有两面。
第一，是架构语义不同： 传统计算机擅长“按规则执行”，NC 更有潜力擅长“在复杂分布上运行”。
 第二，是编程语义不同： 传统计算机主要通过代码安装能力；NC 则更可能通过任务描述、示范、交互轨迹和约束来安装能力。变化不在于代码会消失，而在于代码不再是唯一入口。
在工程上，最先要站稳的不是抽象上的完备性，而是三条可以持续验收的线。
安装—复用： 能力装进去之后，能否在后续任务里稳定复用，而不是每次重新临场发挥？
 失败信号：做法沉淀不下来，每次都靠提示词重新推导。
执行一致性： 同版本、同输入下，结果能不能稳定复现，而不是没有显式更新却悄悄漂移？
 失败信号：没有任何显式能力更新，行为却悄悄变了。
更新治理： 能力变化是否只来自显式更新，而且能被追踪、能回退？
 失败信号：出了问题找不到“到底改了什么”，也回不到上一个稳定版本。
这三条验收线最终会收敛成两道硬门槛：行为一致性与通用可编程。前者决定它能不能部署，后者决定它能不能被称为通用计算机。
要让这些条件在工程上变得可操作，还必须把三条边界讲清：
 状态边界——什么会被持久化并复用，什么只是临时上下文；
 更新边界——什么算运行，什么算重编程；
 证据边界——什么能够被回放、比较与回退。
最直观的类比是数据库事务：运行像读，不该偷偷改写持久能力；更新像写，必须显式、可版本化、可回退。没有这三条边界，运行 / 更新分离就仍然只是口号；有了它们，学习型运行时才开始接近“计算机”，而不只是“会做事的系统”。

3）原型与载体：为什么当前原型长成这样，它已经证明了什么？
先说明一个容易误解的点：我们现在选择视频模型，不是因为它就是 NC 的终局形态。更实际的原因是，它是目前最快能把像素、动作和时序统一进一个端到端原型的路径。在这个意义上，视频模型更像桥，不像房子。对 NC 而言，关键不是“视频更炫”，而是它天然把屏幕帧、动作条件和时间展开模拟放进同一个建模框架里，让我们第一次能用较统一的方式，在命令行（CLI）和图形界面（GUI）两类界面上搭建学习型运行时的原型。
NC 确实借用了 world model 的很多启发：状态随时间演化、动作条件控制、潜在动态等等。但它要解决的不是“世界会怎样演化”，而是“机器该怎样运行”。前者重在预测与推演，后者重在执行、编程与治理。关键就在这里：计算机环境不是一个普通环境，它天然带着可编程性、状态契约和更新边界这些额外约束。
如果按前面的三条工程验收线，再加上一条“是否值得迁移”的净收益要求来看，当前结果可以压成四个判断。
安装—复用：入口出现，但尚未成立。
 我们已经看到学习型运行时能够承接一部分终端和界面动态；但这还不足以证明“安装一次、后续稳定复用”的能力对象已经成立。
执行一致性：局部成立，但只限受控环境。
 受控环境中的 GUI 控制已经逼近可用门槛；但局部一致，不等于长时程一致，受控环境中的稳定，也不等于开放环境中的可靠完成。
更新治理：轮廓出现，但仍有明显缺口。
 今天的原型已经有了运行时的轮廓，但还没有真正具备计算机级的治理接口。版本化更新日志、可审计差分、稳定回放与回滚，都还没有形成完整能力。
净收益：内部有信号，但整体尚未成立。
 现在能看到的，只是 NC 内部训练范式的一些效率信号；但这还不足以证明，相对现有混合路线，NC 已经拿到了整体上的结构性优势。另一面，CLI 场景里的符号能力依然偏弱，离通用可编程运行时的可用门槛还很远。
结论很清楚：视频模型足够让 NC 从概念进入原型，但还不足以让 CNC 从原型进入成立。现有 NC 已经开始承接一些基础运行能力，尤其是 I/O 对齐和短时程控制；但从原型走向通用运行时，最大的缺口仍然是复用、治理、一致性与符号处理。
这里有一个必须反复强调的边界：理论上能表达，不等于工程上能稳定用。很多系统在短时程里看起来连贯，一旦进入组合深度和长时程，漂移就会迅速放大。这个差距不能被放在论文角落，它就是核心问题本身。

4）从 NC 到 CNC：还缺什么，什么条件满足才算真正成立？
在四个对象里，传统计算机定义了执行底座，world model 定义了推演底座，agent 定义了行动接口；NC 争夺的，则是运行时底座本身。它要改写的不是应用层能力，而是“运行中的机器”这一层。
这条路可以只看三个阶段。
Prototype NC： 先把“跑起来”做实。
 这一阶段要证明的是：学习型运行时能稳定承接 I/O 与局部控制。今天的原型已经走到了这里，所以 NC 不再只是想象，而开始有了工程对象的轮廓。
Governed NC： 再把“可治理”做实。
 这一阶段真正要成立的，不只是 demo，而是运行 / 更新分离。能力对象要能被安装、复用、版本化、记录更新日志、比较差异、回放过程并回到上一个稳定版本。没有这些，系统再聪明，也更像一位很会临场发挥的演员，而不是一台可以治理的计算机。
CNC： 最后才把“可替代”做实。
 只有在前两者成立后，才有资格谈通用可编程、行为一致性和相对传统栈的净收益。更机器原生的神经架构，也是在这个阶段才真正成为硬约束：不是更像大脑，而是更像机器；离散操作、组合结构和可验证执行，需要在神经系统里原生成立。
没有前两阶，谈 CNC 都是空话。
这条路最硬的一关，其实正是：运行 / 更新分离并不是神经网络的内生属性。梯度下降天然会把“使用”和“改变自己”缠在一起；NC 要做的，是把这条边界硬生生设计进运行时。否则系统看起来在运行，实际上却在偷偷改写持久能力。
所以我现在的判断是：NC 已经从想象进入原型，CNC 还远未到来。真正决定它会不会成为下一代计算范式的，不是下一次 demo 是否更华丽，而是安装—复用、运行 / 更新分离、回放 / 回滚、符号稳定性和神经净收益，能否在未来几年持续兑现。

5）如果 NC 成立，软件、硬件与“程序”本身都会改变
如果有一天，能力安装、行为一致性和显式更新真的都变成了运行时契约，那么变化就不会只是 agent 更强或 world model 更准，而是四个对象之间的分工会被重新划线，“电脑”这个词本身也会被改写。
传统计算机仍然负责确定性执行，world model 仍然负责推演环境动态，agent 仍然负责在现实工作流中行动；但运行时本身，将第一次从程序栈的一部分，变成学习系统的一部分。
第一，程序对象会变。
 今天我们说“安装软件”，装的是代码；说“写程序”，写的是指令序列。NC 的编程语义成立之后，这个边界会开始移动。能力本身——而不只是实现能力的代码——会变成可安装、可复用、可版本化的一等对象。
第二，架构范式会变。
 CNC 需要的，未必是更像大脑的网络，而更可能是更像机器的神经系统。离散操作、组合结构、显式路由和可验证执行，需要在神经系统里原生成立。通向 CNC 的路，更可能像重新设计计算机，而不是把大脑模拟得更逼真。
第三，电脑的物理边界会松动。
 “这台电脑在哪里”这个问题，答案会越来越不重要。NC 的运行时不需要固定在本地设备上。算力可以远端，可以分布，用户通过接口调用一个持续运行的学习系统，就像今天调用云服务一样；但这一次不只是存储和算力上云——运行时本身、能力本身、行为契约本身都在远端常驻。本地设备会越来越像接口，而不是计算机本身。
第四个系统对象是否真的成立，不看 demo 多惊艳，只看三件事：能力能不能安装，行为能不能稳定，更新能不能治理。
一旦这三件事成立，变化就不再是 AI 会不会更强，而是“电脑”这个词本身要不要重写。
